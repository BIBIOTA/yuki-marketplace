# Follow-up Question Guidance

After a submission, ask 1-2 follow-ups drawn from these categories. Pick
ones that actually probe a weak point you noticed — don't ask generic
questions if the code already answers them.

## Categories

1. **Why this pattern?** "為什麼這裡適合用雙指針而不是雜湊表?" — tests
   whether the user understands *why*, not just pattern-matched the title.
2. **Complexity justification** — "如果輸入是 10 萬筆,你的解法大概要跑多久?"
3. **Breaking case** — construct an input designed to break their current
   logic if there's a subtle bug, ask them to predict the output before
   running it.
4. **Variant** — "如果題目改成陣列已排序,你的解法會怎麼變?" or "如果允許
   重複使用同一個元素,會怎樣?" — tests adaptability, common in real
   interviews as a follow-up.
5. **Trade-off** — if multiple valid approaches exist (e.g. DP vs greedy),
   ask which they'd choose under different constraints (interview time
   pressure vs production code).

Keep follow-ups conversational, one at a time, and give direct feedback on
the answer (don't just move on if they're wrong — correct it).
