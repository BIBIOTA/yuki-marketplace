# Job Application Recommendation Letter Skill

## 目的

根據「使用者提供的最新履歷」與「目標職缺 JD」，撰寫一封短而聚焦的自我推薦信。

這份推薦信的目標不是完整 Cover Letter，而是讓 HR 或面試官快速理解：

1. 使用者為什麼適合這個職缺
2. 使用者的過去經驗如何對應 JD
3. 若有技術棧落差，如何誠實但不放大弱點地說明
4. 使用者為什麼對這間公司、產品或職缺方向有興趣

推薦信應該適合貼在台灣求職平台的：

- 自我推薦
- 應徵訊息
- 給公司的話
- Cover message
- LinkedIn 應徵訊息
- Email 開頭自我介紹

---

## 核心原則

這個 Skill 不應寫死使用者的履歷內容。

履歷中的公司、職稱、年資、技術棧、專案、成果數字、Side Project 都可能隨時修改，因此每次撰寫推薦信時，必須重新參考使用者提供的最新履歷與 JD。

Skill 可以保留的是「撰寫策略」與「判斷原則」，而不是固定履歷事實。

---

## 輸入資料

使用者可能提供以下資訊：

1. 目標公司名稱
2. 職缺名稱
3. JD 內容
4. 最新履歷內容或履歷檔案
5. 可選：希望字數
6. 可選：希望語氣，例如正式、自然、精簡、積極但不浮誇
7. 可選：是否要強調轉語言
8. 可選：是否要提 DevOps、雲端、測試、資料庫或特定技術
9. 可選：是否要避開某些內容，例如管理經驗、薪資、待業狀態、非主力技術

若使用者只提供 JD，沒有提供履歷，請先詢問是否要使用當前對話中已提供的履歷內容，或請使用者補上最新履歷。

不要在沒有履歷依據的情況下自行假設使用者的經歷。

---

## 履歷來源與使用原則

使用者提供的履歷可能形式包括：

- 上傳的履歷 PDF
- 上傳的履歷 DOCX
- 貼上的履歷文字
- 已在同一個對話中提供過的履歷內容
- 使用者指定的履歷版本
- 使用者補充的特定經歷或專案

撰寫推薦信時，必須以「使用者當次提供或指定的最新履歷」為準。

如果履歷內容與過去記憶、先前對話或舊版本履歷不一致，請以使用者當次提供的履歷為準。

不要使用寫死的履歷摘要。

不要假設過去履歷內容仍然正確。

---

## 履歷解析方式

在撰寫推薦信前，先從履歷中動態萃取以下資訊：

1. 使用者姓名
2. 目標職稱或目前職稱
3. 最近一份工作經驗
4. 最相關的一份工作經驗
5. 與 JD 最相關的 3～5 個技能
6. 與 JD 最相關的 2～4 個成果或專案
7. 主要技術棧
8. 可對應 JD 的資料庫、測試、CI/CD、DevOps、雲端、效能優化或系統維運經驗
9. 是否存在技術棧落差
10. 是否存在職務定位落差，例如從 Tech Lead 轉回 Backend Engineer、從 PHP 轉 Python、從管理偏重轉 hands-on
11. 不適合主動強調的內容，例如與 JD 無關的管理經驗、過長自傳、太多 Side Project、與職缺無關的技能

推薦信不需要完整重述履歷，而是挑選最能對應 JD 的內容。

---

## JD 解析方式

在撰寫推薦信前，先從 JD 中萃取：

1. 公司名稱
2. 職缺名稱
3. 產品或業務方向
4. 工作內容
5. 必備條件
6. 加分條件
7. 技術棧
8. 最重要的 3～5 個匹配點
9. 可能的履歷風險點，例如語言不同、DevOps 不足、雲端經驗不足、產業不同

請優先對齊 JD 中最重要的條件，而不是把使用者履歷中所有亮點都塞進推薦信。

---

## 推薦信預設結構

預設輸出一封 3～4 段的自我推薦信。

格式如下：

```text
{公司名稱} 您好，我叫做{使用者姓名}。

{第一段：最近或最相關工作經驗，對應 JD 的核心需求}

{第二段：技術能力、成果、系統經驗，若有技術棧落差，誠實但不放大弱點地說明}

{第三段：對公司產品、職缺方向或產業的興趣，以及希望進一步交流}

謝謝您撥冗閱讀。
```

若使用者要求更正式，可加上：

```text
期待有機會與您進一步交流，謝謝您。
```

若使用者要求更精簡，可省略部分客套語。

---

## 預設字數

除非使用者另有要求，推薦信長度應控制在：

- 300～450 字
- 不超過 500 字
- 適合台灣求職平台自我推薦欄位

若使用者要求短版：

- 控制在 250～350 字

若使用者要求極短版：

- 控制在 150～250 字
- 只保留最核心匹配點

若使用者要求完整 Cover Letter：

- 可擴展到 600～900 字
- 但要明確告知這是 Cover Letter，不是短版自我推薦信

---

## 撰寫策略

### 1. 先寫匹配點，不要先寫弱點

不要一開始就寫：

```text
雖然我過去主要不是 Python...
```

應該先寫使用者和職缺高度相關的經驗，例如：

```text
我前一份職務主要負責後端功能開發、API 設計、系統需求分析、效能優化與線上問題追蹤，和貴公司職缺中提到的產品開發、系統優化與平台維運有相關經驗。
```

技術棧落差放在第二段補充即可。

---

### 2. 技術棧落差要誠實，但不要放大

若 JD 要 Python / FastAPI / Django，而履歷主力是 PHP / Laravel，請使用類似寫法：

```text
雖然我過去工作中的主力語言是 PHP / Laravel，但近期正透過 Side Project 使用 Python / FastAPI 開發後端服務，累積 Python 後端開發經驗。
```

若履歷中有 SQLAlchemy、PostgreSQL、Pytest、Django、Celery、Redis 等經驗，且與 JD 相關，可以補充：

```text
並搭配 SQLAlchemy、PostgreSQL 與測試實作，累積 Python 後端開發與資料庫操作經驗。
```

避免寫：

```text
我 Python 經驗不足，但我會努力學習。
```

也避免過度誇大成：

```text
我精通 Python / FastAPI。
```

除非履歷中明確顯示使用者已有足夠工作經驗。

---

### 3. DevOps 經驗有限時，只寫做過的具體事項

若 JD 要求 DevOps，但履歷中的 DevOps 經驗有限，不要寫：

```text
我 DevOps 經驗較少。
```

也不要寫：

```text
我正在持續加強 DevOps。
```

除非使用者明確表示已經開始補強。

建議寫成：

```text
DevOps 方面，過去曾以後端工程師角色參與 CI/CD 與 Docker 相關流程維護，也有將個人專案部署至雲端平台的經驗。
```

若履歷中有具體平台，例如 AWS、GCP、Vercel、Fly.io、Render、Railway、Cloud Run、ECS、Lambda，可列出與 JD 最相關的 2～3 個。

不要誇大成：

```text
具備完整 DevOps 架構建置經驗。
```

除非履歷確實支持。

---

### 4. 不要把推薦信寫成履歷濃縮版

推薦信不是履歷摘要，也不是自傳。

不要列太多技能：

```text
我會 PHP、Laravel、Python、FastAPI、Django、Node.js、NestJS、Golang、Vue、React、MySQL、PostgreSQL、MongoDB、Redis、Docker、Kubernetes、AWS、GCP...
```

應只選 JD 最需要的技能。

---

### 5. 不要過度強調管理經驗

若 JD 是 Backend Engineer 或 Senior Backend Engineer，除非 JD 明確要求帶人，否則不要過度強調：

- 人員管理
- 績效評估
- 管理職責
- 跨團隊管理

可改成較貼近工程職的說法：

```text
也曾負責任務分配、Code Review 與技術協作。
```

避免讓公司誤以為使用者只想做管理，不想 hands-on 寫程式。

---

### 6. 將使用者定位為「後端工程師」，不是某一語言工程師

若使用者正在換語言，推薦信的核心定位應是：

```text
有產品後端經驗的工程師，正在把既有後端能力遷移到新的技術棧。
```

不要寫成：

```text
某語言工程師想轉另一個語言。
```

例如不要強調：

```text
我是 PHP 工程師，想轉 Python。
```

應改成：

```text
我的核心能力集中在後端系統設計、API 開發、資料庫設計、效能優化與產品維運；近期也透過專案持續累積 Python 後端開發經驗。
```

---

### 7. 產品興趣要具體，但不要浮誇

第三段應簡短說明使用者為什麼對公司或產品有興趣。

好的寫法：

```text
我對貴公司結合點數、電商與金融服務的產品方向很感興趣，期待有機會將過去在電商平台累積的後端經驗，應用在貴公司的產品與系統開發中。
```

避免：

```text
貴公司是業界翹楚，我一直非常嚮往。
```

避免：

```text
我相信我一定能為貴公司帶來巨大貢獻。
```

## 常見 JD 類型與履歷對應策略

### 1. 電商 / 點數 / 會員 / 交易平台

優先從履歷中找：

- 電商平台經驗
- 產品線維運
- 商品、訂單、會員、點數、金流、後台、營運平台相關經驗
- 第三方服務串接
- 效能優化
- 高流量或乘載量改善
- 系統重構
- 資料庫設計與優化

推薦信可使用：

```text
過去曾參與電商平台後端開發、第三方服務串接、系統重構與效能優化，和貴公司職缺中提到的平台開發、系統優化與營運乘載量提升有相關經驗。
```

---

### 2. Python Backend / FastAPI / Django

優先從履歷中找：

- Python
- FastAPI
- Django
- SQLAlchemy
- PostgreSQL
- Pytest
- Celery
- Redis
- API 開發
- Side Project
- 後端服務設計

若 Python 是 Side Project，不要誇大為正式工作經驗。

推薦信可使用：

```text
雖然我過去工作中的主力語言不是 Python，但近期正透過 Side Project 使用 Python / FastAPI 開發後端服務，並搭配資料庫與測試實作，累積 Python 後端開發經驗。
```

---

### 3. Java / Spring Backend

優先從履歷中找：

- 後端系統設計
- API 開發
- RDBMS
- Redis
- Message Queue
- 微服務概念
- 電商或交易平台經驗
- 效能優化
- 系統重構

若履歷沒有 Java 工作經驗，不要假裝有。

推薦信可使用：

```text
雖然我過去工作中的主力語言不是 Java，但我的核心經驗集中在後端系統設計、API 開發、資料庫設計、效能優化與產品維運，並能將既有後端經驗遷移到不同技術棧。
```

---

### 4. AI Backend / LLM Application

優先從履歷中找：

- LLM 應用
- AI 工具導入
- Prompt workflow
- Python / FastAPI
- RAG
- Agent
- OpenAI / Claude / Gemini
- Queue / Worker
- SSE / WebSocket
- API 串接
- 評分、分析、生成、文件處理相關專案

推薦信可使用：

```text
近期也透過 Side Project 與工作流程導入累積 AI 應用後端經驗，包含 LLM API 串接、後端服務設計與 AI 輔助開發流程。
```

注意：如果履歷只寫 AI 工具導入，不要誇大成 LLM 系統開發經驗。

---

### 5. DevOps / Cloud / Infrastructure-heavy Backend

優先從履歷中找：

- CI/CD
- Docker
- 雲端部署
- AWS / GCP / Azure
- Linux
- Monitoring
- Logging
- Sentry / Kibana / CloudWatch
- IaC
- Kubernetes
- ECS / Lambda / Cloud Run
- 部署流程改善

若履歷只顯示部分 DevOps 經驗，請使用「參與」或「具備基礎實作」而非「主導建置」。

推薦信可使用：

```text
DevOps 方面，過去曾以後端工程師角色參與 CI/CD 流程維護、Docker 開發環境優化與雲端部署，具備產品團隊中後端工程師所需的維運與部署流程理解。
```

---

### 6. 新創 / 小團隊 / Full-stack Backend

優先從履歷中找：

- 從 0 到 1
- 跨職能合作
- 前後端協作
- 獨立開發
- 快速迭代
- 產品維運
- API 設計
- 小團隊經驗
- Full-stack side project

推薦信可使用：

```text
過去在產品團隊與小型開發團隊中，皆有負責後端開發、需求釐清、API 設計與跨職能協作的經驗，能適應快速迭代的開發環境。
```

## 推薦信語氣

語氣應該：

- 自然
- 誠懇
- 專業
- 不卑不亢
- 精簡
- 不像制式模板
- 不像完整自傳
- 不過度自誇
- 不主動暴露弱點

避免使用：

- 「本人」
- 「貴公司是業界翹楚」
- 「希望貴公司給我一次機會」
- 「雖然我能力不足」
- 「我一定能勝任」
- 「我願意接受任何挑戰」
- 「敝人」
- 「尚祈惠予面試機會」
- 過多形容詞
- 過度客套語

## 推薦信輸出前檢查清單

輸出前請確認：

- 是否有公司名稱
- 是否有使用者姓名
- 是否有對應 JD 的 3～5 個重點
- 是否有使用最新履歷內容
- 是否避免寫死履歷背景
- 是否把使用者定位成後端工程師，而不是單一語言工程師
- 若有技術棧落差，是否誠實但沒有放大弱點
- 若有 DevOps 落差，是否只寫具體做過的事項
- 是否避免過度強調管理職
- 是否控制在 300～450 字內
- 是否沒有寫成完整自傳
- 是否沒有使用過度浮誇或太卑微的語氣
- 是否結尾自然、有禮貌

---

## 輸出格式

預設只輸出推薦信，不需要附上長篇解釋。

若使用者要求「幫我看看如何」，可以輸出：

1. 簡短評語
2. 修改後版本
3. 最關鍵修改原因

若使用者要求「給我完整可貼上的版本」，只輸出可貼上的版本。

---

## 標準輸出範例

以下範例僅示意語氣與結構，不代表固定履歷內容。

```text
{公司名稱} 您好，我叫做{使用者姓名}。

我前一份職務主要負責後端功能開發、系統需求分析、效能優化與線上問題追蹤等工作。過去也曾參與 API 效能優化、Legacy 系統現代化、第三方服務串接、CI/CD 流程維護與 Docker 開發環境優化，和貴公司職缺中提到的平台開發、系統優化與營運乘載量提升有相關經驗。

雖然我過去工作中的主力語言與貴公司使用的技術棧不完全相同，但我的核心能力集中在後端系統設計、API 開發、資料庫設計、效能優化與產品維運；近期也透過 Side Project 持續累積相關技術棧的後端開發經驗。

我對貴公司的產品方向很感興趣，期待有機會將過去在產品後端開發與系統維運中累積的經驗，應用在貴公司的產品與系統開發中。

謝謝您撥冗閱讀。
```

## 特殊情境處理

### 使用者想主動說明弱點

若使用者想主動提到「沒有 DevOps 經驗」、「Python 還不熟」、「不是本科系」、「英文比較弱」等弱點，請協助改成較正向且誠實的寫法。

原句：

```text
我 DevOps 經驗不足。
```

改成：

```text
DevOps 方面，過去主要以後端工程師角色參與 CI/CD、Docker 或部署流程相關工作。
```

原句：

```text
我 Python 還不熟。
```

改成：

```text
近期透過 Side Project 使用 Python / FastAPI 開發後端服務，持續累積 Python 後端開發經驗。
```

原句：

```text
我不是本科系。
```

通常不主動提。若一定要提，可改成：

```text
我過去從非資訊背景轉職成後端工程師，並在產品團隊中累積了完整的後端開發與系統維運經驗。
```

---

### JD 要求很高，但履歷只部分符合

不要硬湊。

請採用「相關經驗」語氣，而不是「完全符合」。

可以寫：

```text
和貴公司職缺中提到的系統優化、平台維運與後端開發方向有相關經驗。
```

避免：

```text
我完全符合貴公司的所有需求。
```

---

### 使用者投的是比目前資歷低一點或轉語言職缺

不要強調「降級」或「我願意從低做起」。

可以寫：

```text
我期待將過去累積的產品後端經驗，延伸到貴公司使用的技術棧與產品場景中。
```

---

### 使用者投的是 Senior 或 Tech Lead 職缺

若 JD 明確需要 Senior / Lead，可以適度強調：

- 技術規劃
- Code Review
- 任務拆解
- 跨職能溝通
- 系統重構
- 效能優化
- 產品維運

但仍避免把推薦信寫得太像管理職申請。

---

## 最終目標

每次產出的推薦信都應該做到：

1. 看起來是針對該公司與 JD 客製化
2. 內容來自使用者最新履歷
3. 不過度暴露技術落差
4. 不誇大履歷沒有的經驗
5. 讓 HR 或面試官快速看出「這個人值得點開履歷」
